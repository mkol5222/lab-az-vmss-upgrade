VMSS1 → VMSS2 Migration & Cost Teardown

Safely retire the legacy R8120 CloudGuard VMSS after cutting traffic over to the new R82 VMSS — while preserving the shared VNet environment.

Project: lab-az-vmss-upgrade Region: westeurope VNet: labvmssup-vnet (10.101.0.0/16) 11 resources to destroy network layer kept

Generated 2026-06-12 · Strategy: targeted terraform destroy · Method A (config drift accepted)

1. Overview

vmss2/ deploys a new R82 CloudGuard scale set with its own internal Load Balancer into the existing network created by vmss1/. Once routing is cut over to the new firewalls, the legacy vmss1/ compute and Load Balancers become redundant and continue to incur cost.

This report describes a safe, targeted teardown that removes only the billable vmss1/ resources (scale set, two Load Balancers, public IP, boot-diagnostics storage) while keeping the elementary network environment (VNet, subnets, NSG, route tables) that both vmss2/ and vmss1-linux/ depend on.

vmss1 (legacy R8120)
VMSS + 2× Standard LB + public IP → destroy
→
vmss2 (new R82)
VMSS + internal LB → becomes the active NVA
→
labvmssup-vnet
VNet + subnets + NSG + route tables → keep (shared)
Why not split the VNet into its own state? The CloudGuard module (module.example_module) is monolithic — it provisions the VNet, subnets, Load Balancers and the VMSS together. A targeted destroy is the lowest-risk lever. The trade-off: the vmss1/ configuration still declares the destroyed resources, so its state will show drift. Do not run make vmss1 again, or the resources will be recreated.

2. Critical dependency — why the network layer must stay

  • vmss2/ references the VNet and both subnets as Terraform data sources (read-only) — it does not manage or recreate them.
  • The new vmss2 firewall instances run inside vmss1’s Backend subnet (10.101.2.0/24).
  • That subnet is associated with vmss1’s NSG (labvmssup-vmss1-nsg) and vmss1’s Backend route table, both owned by the vmss1/ state.
Do not destroy the VNet, subnets, NSG, or route tables. Removing them would strip the NSG and route table off the subnet where the vmss2 firewalls live, breaking the new deployment. These resources are also free or negligible in cost — all real savings come from the VMSS, Load Balancers and public IP.

3. Safe migration strategy

  1. Confirm vmss2 is up Ensure the new scale set is deployed and VMSS2_GW is populated in the root .env. Run make vmss2 if needed (its up.sh captures the backend LB frontend IP).
  2. Repoint the Linux diagnostics subnet to vmss2 Switch the default gateway of linux-subnet from vmss1’s NVA to vmss2’s, then re-apply.
  3. Verify connectivity through vmss2 SSH into the Linux VM (make linux-ssh) and confirm egress now flows through the new R82 firewalls (e.g. traceroute / curl to the internet) before removing anything.
  4. Dry-run the teardown Run a terraform plan -destroy with the target list and review that only the intended compute/LB resources are scheduled for deletion.
  5. Execute the targeted destroy Apply the terraform destroy -target=… command (Section 6).
  6. Post-migration verification Confirm the VNet/subnets/NSG/route tables are intact, vmss1 VMSS+LBs are gone, and cd vmss2 && terraform plan reports no changes.

7. Rollback

The vmss1/ configuration still declares every destroyed resource. To recreate the legacy deployment, simply re-run make vmss1 (a normal terraform apply). Note this restores the duplicate firewall fleet and its cost — only do so if the migration must be reverted.

8. Cost-savings summary

Removed (vmss1)

  • 2× Standard_D4ds_v5 firewall instances
  • 2× Standard Load Balancers
  • 1× Standard public IP
  • 1× boot-diagnostics storage account

Remaining (vmss2 + shared)

  • 2× Standard_D4ds_v5 R82 instances
  • 1× internal Load Balancer
  • Shared VNet, subnets, NSG, route tables (free / negligible)
  • Linux diagnostics VM (Standard_B1s)

Net effect: the duplicate firewall fleet and its Load Balancers are eliminated, roughly halving the CloudGuard compute and load-balancing spend, with the new R82 deployment as the sole active NVA.
